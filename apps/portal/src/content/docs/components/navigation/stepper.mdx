The `Stepper` component provides a vertical progress indicator for multi-step workflows like wizards and onboarding flows. It supports dynamic substeps, multiple visual states, transition animations, navigation guards, and keyboard navigation.

It is composed of `Stepper.Root`, `Stepper.Step`, and `Stepper.SubStep` to offer a structured and customizable interface.

<DocsPage.ComponentExample
  client:only
  code={`() => {
    const [step, setStep] = React.useState("connect");

    return (
        <div className="w-[300px]">
            <Stepper.Root value={step} onValueChange={setStep} title="Pipeline Setup">
                <Stepper.Step value="connect" title="Connect" description="Choose your provider" />
                <Stepper.Step value="configure" title="Configure" description="Set up your pipeline" />
                <Stepper.Step value="review" title="Review" description="Confirm and create" />
            </Stepper.Root>
        </div>
    );

}`}
/>

## Usage

```typescript jsx
import { Stepper } from "@harnessio/ui/components";

//...

const [currentStep, setCurrentStep] = useState("step-1");

return (
  <Stepper.Root value={currentStep} onValueChange={setCurrentStep} title="Setup">
    <Stepper.Step value="step-1" title="First Step" description="Do something" />
    <Stepper.Step value="step-2" title="Second Step" description="Do another thing" />
    <Stepper.Step value="step-3" title="Final Step" description="All done" />
  </Stepper.Root>
)
```

## Anatomy

All parts of the `Stepper` component can be imported and composed as required.

```typescript jsx
<Stepper.Root>
  <Stepper.Step>
    <Stepper.SubStep />
    <Stepper.SubStep />
  </Stepper.Step>
  <Stepper.Step />
</Stepper.Root>
```

## Step States

Steps display different visual states based on their position relative to the active step and explicit state overrides.

<DocsPage.ComponentExample
  client:only
  code={`() => {
    const [step, setStep] = React.useState("active");

    return (
        <div className="w-[300px]">
            <Stepper.Root value={step} onValueChange={setStep} title="Step States">
                <Stepper.Step value="completed" title="Completed" description="This step is done" />
                <Stepper.Step value="active" title="Active" description="Currently on this step" />
                <Stepper.Step value="upcoming" title="Upcoming" description="Not yet reached" />
                <Stepper.Step value="error-step" title="Error" description="Something went wrong" state="error" />
            </Stepper.Root>
        </div>
    );

}`}
/>

<Aside type="tip">
  Step states are derived automatically based on position (completed for steps
  before active, upcoming for steps after), but can be explicitly overridden
  with the `state` prop to show `error` or `skipped`.
</Aside>

## Loading and Blocking

Steps can show a loading spinner and optionally block forward navigation while an async operation is in progress.

<DocsPage.ComponentExample
  client:only
  code={`() => {
    const [step, setStep] = React.useState("checking");

    return (
        <div className="w-[300px]">
            <Stepper.Root value={step} onValueChange={setStep} title="Connectivity Check">
                <Stepper.Step value="provider" title="Select Provider" description="Choose VCS" />
                <Stepper.Step
                    value="checking"
                    title="Checking Connection"
                    description="Verifying access..."
                    loading
                    blocking
                />
                <Stepper.Step value="repo" title="Select Repository" description="Pick your repo" />
            </Stepper.Root>
        </div>
    );

}`}
/>

<Aside type="note">
  When `blocking` is set on the active step, navigation to subsequent steps is
  prevented. The user can still navigate backwards. Combine with `loading` to
  show a spinner on the indicator.
</Aside>

## SubSteps

Steps can contain dynamic substeps that appear when the parent step is active. SubSteps support conditional rendering based on application state.

<DocsPage.ComponentExample
  client:only
  code={`() => {
    const [step, setStep] = React.useState("auth");

    return (
        <div className="w-[300px]">
            <Stepper.Root value={step} onValueChange={setStep} title="GitHub Integration">
                <Stepper.Step value="connect" title="Connect" description="Choose provider">
                    <Stepper.SubStep value="auth" title="Authenticate" description="OAuth with GitHub" />
                    <Stepper.SubStep value="test" title="Test Connection" description="Verify access" />
                </Stepper.Step>
                <Stepper.Step value="configure" title="Configure" description="Set up pipeline" />
                <Stepper.Step value="done" title="Done" description="All set" />
            </Stepper.Root>
        </div>
    );

}`}
/>

<Aside type="tip">
  SubSteps are conditionally rendered — they only appear when their parent step
  is active. The `value` passed to `Stepper.Root` can be either a step value or
  a substep value.
</Aside>

## Navigation Guard

Use the `onBeforeChange` prop to confirm navigation (e.g., warn users about losing progress when going back).

<DocsPage.ComponentExample
  client:only
  code={`() => {
    const [step, setStep] = React.useState("configure");
    const steps = ["connect", "configure", "review"];

    return (
        <div className="w-[300px]">
            <Stepper.Root
                value={step}
                onValueChange={setStep}
                title="With Guard"
                onBeforeChange={(from, to) => {
                    const fromIdx = steps.indexOf(from);
                    const toIdx = steps.indexOf(to);
                    if (toIdx < fromIdx) {
                        return "Going back will reset your progress. Are you sure?";
                    }
                    return true;
                }}
            >
                <Stepper.Step value="connect" title="Connect" description="Choose provider" />
                <Stepper.Step value="configure" title="Configure" description="Set up pipeline" />
                <Stepper.Step value="review" title="Review" description="Confirm settings" />
            </Stepper.Root>
        </div>
    );

}`}
/>

<Aside type="note">
  `onBeforeChange` receives `(fromValue, toValue)` and should return: - `true`
  to allow navigation - `false` to silently block navigation - A `string` to
  show a confirmation dialog with that message
</Aside>

## API Reference

### `Root`

The root container that manages stepper state, provides context to children, and renders the header with title and progress counter.

```typescript jsx
<Stepper.Root
  value={currentStep}            // active step or substep value
  onValueChange={setStep}        // callback when step changes
  title="Setup"                  // [OPTIONAL] displayed in the header
  completed={false}              // [OPTIONAL] shows "Complete" instead of counter
  showConnectors={true}          // [OPTIONAL] show connector lines between steps
  onBeforeChange={guardFn}       // [OPTIONAL] navigation guard
  skeletonCount={3}              // [OPTIONAL] skeleton items when no children
  className="my-class"           // [OPTIONAL] custom class on the nav element
>
  {/* Step components */}
</Stepper.Root>
```

  loading={false}                // [OPTIONAL] show spinner on indicator
  blocking={false}               // [OPTIONAL] block forward navigation
  disabled={false}               // [OPTIONAL] prevent interaction

  {/* Optional Stepper.SubStep children */}

## Accessibility

The stepper implements the following accessibility features:

- **ARIA**: Uses `aria-current="step"` on the active step, `aria-label` on each step button with position context, and `aria-disabled` for disabled steps
- **Live Region**: Announces step changes via `aria-live="polite"` for screen readers
- **Keyboard Navigation**: Supports ArrowDown, ArrowUp, Home, and End keys with roving tabindex
- **Reduced Motion**: Respects `prefers-reduced-motion` by disabling transition animations

## Best Practices

1. **Unique Values**: Every step and substep must have a unique `value` across the entire stepper
2. **Controlled State**: The component is always controlled — provide both `value` and `onValueChange`
3. **Blocking Steps**: Use `blocking` for async operations that must complete before proceeding (e.g., connectivity checks)
4. **Navigation Guards**: Use `onBeforeChange` to warn about data loss, not to enforce linear progression
5. **SubStep Cleanup**: When substeps unmount (e.g., provider changes), the stepper automatically falls back to the parent step value
6. **Skeleton State**: Pass no children to show a skeleton — useful while step configuration loads asynchronously
