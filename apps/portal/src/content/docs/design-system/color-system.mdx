## Why LCH?

All colors are authored in <span style="font-weight: 600">LCH color space</span> (Lightness, Chroma, Hue). LCH is perceptually uniform — equal numeric steps produce equal visual differences. This makes it straightforward to build consistent color ramps and verify contrast ratios mathematically.

HEX values exist only as a Figma export fallback. <span style="font-weight: 600">LCH is the source of truth</span>.

```json
{
  "gray": {
    "25": { "$value": "lch(99% 0 272)" },
    "50": { "$value": "lch(97% 0 272)" },
    "100": { "$value": "lch(92% 1 272)" },
    "500": { "$value": "lch(65% 6 272)" },
    "900": { "$value": "lch(16% 6 280)" },
    "1000": { "$value": "lch(7.5% 1 280)" }
  }
}
```

Each color family has 14 stops (25–1000) plus `pure.white` and `pure.black`. The full palette includes: gray, red, blue, indigo, forest, yellow, purple, cyan, mint, orange, pink, violet, brown, lime, amber.

## Semantic Levels

Core primitives have no meaning — `gray.500` is just a value. <span style="font-weight: 600">Semantic tokens</span> assign purpose. They're defined per-theme in `design-tokens/mode/{dark,light}/` and map <span style="font-weight: 600">intent</span> to <span style="font-weight: 600">value</span>.

## Color Sets — The Core Pattern

The most important architectural concept. Color sets are <span style="font-weight: 600">pre-built color pairs</span> — coordinated background + text combinations for different usage scenarios. Each pair is verified for WCAG contrast compliance, so developers never need to manually check accessibility.

Every semantic color is defined as a <span style="font-weight: 600">set</span> with three emphasis levels:

```
set.{color}.primary     → bg, bg-hover, bg-selected, text, ring
set.{color}.secondary   → bg, bg-hover, bg-selected, text
set.{color}.outline     → bg, bg-hover, bg-selected, text, border
```

## Component-Specific Colors

Some components need colors that don't fit the generic semantic tokens — gradients, overlays with custom alpha, or domain-specific highlights. These live in the `comp.*` namespace and reference semantic tokens internally:

```json
{
  "comp.alert.fade.danger": "linear-gradient(0deg, {set.danger.secondary.bg} 42%, ...)",
  "comp.dialog.backdrop": "{gray.600} + alpha(0.2)",
  "comp.diff.add-content": "{set.success.secondary.bg} + alpha(0.45)",
  "comp.avatar.shadow": "{gray.1000} + alpha(0.08)"
}
```

Component tokens follow the same theme-switching rules — their source values (`set.*`, `bg.*`, `gray.*`) are resolved per theme, so component colors adapt automatically.

## State Tokens & Color Modifiers

Interactive states use the `$extensions` system from Token Studio to apply LCH-space transformations:

```json
{
  "state": {
    "hover": {
      "$value": "{gray.850}",
      "$extensions": {
        "studio.tokens": {
          "modify": { "type": "alpha", "value": "0.2", "space": "lch" }
        }
      }
    }
  }
}
```

This produces `lch(from var(--cn-gray-850) l c h / 0.2)` — a relative-color alpha overlay. The same modifier system supports `darken` and `lighten` operations for contrast variant themes (dimmer, high-contrast).

### Available Modifiers

Token Studio supports four color modifiers:

- <span style="font-weight: 600">alpha</span> — adjusts opacity (0 =
  transparent, 1 = solid)
- <span style="font-weight: 600">lighten</span> — increases lightness by
  percentage (0.2 = 20% lighter)
- <span style="font-weight: 600">darken</span> — reduces lightness by percentage
  (0.2 = 20% darker)
- <span style="font-weight: 600">mix</span> — blends with another color (0.5 =
  50/50 mix)
